Да — сделаем тебе **рабочую учебную схему**, где ты один раз создаёшь “каркас инструкций”, а дальше **общаешься с Codex в IDE обычными промптами**, и он сам:

* создаёт папки и файлы,
* генерирует роуты,
* пишет тесты,
* чинит баги,
* обновляет архитектуру,
* и может участвовать в GitHub-процессе.

Codex в VS Code действительно рассчитан на такие задачи: он умеет читать, редактировать и запускать код, а skills поддерживаются в CLI, IDE extension и Codex app. Для GitHub у OpenAI есть и отдельная интеграция, и GitHub Action. ([OpenAI Developers][1])

Ниже дам тебе **именно пошаговую учебную инструкцию**, без лишней магии.

---

## Как мы это организуем

У тебя будет 2 слоя:

**Слой 1. Репозиторий и файлы-инструкции**

* `AGENTS.md`
* `SKILLS.md`
* `CONTROL_FLOW.md`
* `architecture.md`
* `testing.md`

**Слой 2. Запросы в Codex IDE**

* “Создай структуру проекта”
* “Сгенерируй модуль tasks”
* “Напиши тесты”
* “Запусти тесты и исправь ошибки”
* “Обнови архитектуру”
* “Подготовь коммит”

Важно: я не буду обещать, что в твоей IDE есть **ровно такие slash-команды** как `/agent create-project`, потому что интерфейс и команды могут отличаться по версии и режиму. Надёжный вариант — использовать **натуральные текстовые инструкции** в чате Codex внутри VS Code. Это согласуется с тем, как Codex IDE extension описан OpenAI. ([OpenAI Developers][1])

# Шаг 1. Создай пустой проект

В терминале VS Code:

```bash
mkdir codex-task-manager
cd codex-task-manager
git init

python -m venv venv
```

Активация окружения:

macOS / Linux:

```bash
source venv/bin/activate
```

Windows:

```bash
venv\Scripts\activate
```

Установи базовые зависимости:

```bash
pip install fastapi uvicorn pytest httpx pytest-cov
pip freeze > requirements.txt
```

Зачем это нужно:

* `fastapi` — API
* `uvicorn` — запуск сервера
* `pytest` — тесты
* `httpx` — удобно для интеграционных тестов
* `pytest-cov` — покрытие тестами

# Шаг 3. Заполни `AGENTS.md`

Этот файл объясняет Codex, **какие роли он должен играть** в проекте.

```md
# AGENTS.md

## Общие правила проекта

Проект написан на Python.
Стек: FastAPI + pytest.
Все комментарии в коде и поясняющие docstring должны быть на русском языке.
Если агент создает новый файл с кодом, он обязан добавлять комментарии и краткие пояснения в ключевых местах.

Проект должен следовать модульной структуре:
- src
- modules
- controllers
- routers
- middleware
- views
- interfaces
- utils
- tests/unit
- tests/integration

Все изменения должны быть согласованы с architecture.md и testing.md.

---

## Agent: Architect

Роль:
Отвечает за архитектуру проекта.

Обязанности:
- создавать и поддерживать согласованную структуру проекта
- обновлять architecture.md при изменении архитектуры
- следить, чтобы новые модули соответствовали общей схеме
- предлагать рефакторинг, если структура разрастается или дублируется код

---

## Agent: Backend

Роль:
Отвечает за реализацию backend-кода.

Обязанности:
- создавать роуты FastAPI
- создавать controllers
- создавать services
- создавать repositories
- создавать interfaces / schemas
- добавлять фильтрацию, пагинацию и валидацию
- писать комментарии на русском языке

---

## Agent: Testing

Роль:
Отвечает за тестирование.

Обязанности:
- создавать unit tests
- создавать integration tests
- добавлять или обновлять тесты после изменений в коде
- запускать тесты и исправлять несовместимости между кодом и тестами
- следить за покрытием не ниже целевого значения из testing.md

---

## Agent: Maintainer

Роль:
Отвечает за сопровождение проекта.

Обязанности:
- находить баги
- исправлять failing tests
- поддерживать README, architecture.md и testing.md в актуальном состоянии
- готовить осмысленные git commit message
- при необходимости готовить изменения для GitHub Actions
```

### Что делает этот файл

Он задаёт Codex не просто одну задачу, а **режим мышления по ролям**.
Это полезно, потому что ты можешь дальше говорить не просто “сделай код”, а, например:

> “Действуй как Agent: Backend и создай модуль tasks”

или

> “Действуй как Agent: Testing и допиши тесты для фильтрации”

# Шаг 4. Заполни `SKILLS.md`

Этот файл описывает **повторяемые навыки**, которые Codex должен применять.

```md
# SKILLS.md

## Skill: generate_project_structure

Описание:
Создает структуру Python-проекта по архитектурной схеме.

Что должен сделать:
- создать папки src, modules, controllers, routers, middleware, views, interfaces, utils
- создать папки tests/unit и tests/integration
- создать базовые __init__.py там, где это уместно
- не создавать лишние файлы без необходимости

---

## Skill: generate_module

Описание:
Создает новый модуль приложения по имени сущности.

Что должен сделать:
- создать repository
- создать service
- создать controller
- создать router
- создать schemas / interfaces
- зарегистрировать модуль в приложении
- добавить комментарии на русском языке

Пример:
tasks, users, auth

---

## Skill: generate_crud_routes

Описание:
Создает CRUD-роуты для сущности.

Что должен сделать:
- GET list
- GET by id
- POST create
- PATCH update
- DELETE remove
- предусмотреть фильтрацию и пагинацию там, где это уместно

---

## Skill: add_filtering

Описание:
Добавляет фильтрацию в список сущностей.

Пример:
- status
- completed
- search
- created_from
- created_to

---

## Skill: add_pagination

Описание:
Добавляет пагинацию в list endpoint.

Требования:
- page
- limit
- total
- items
- в тестах проверить корректную разбивку

---

## Skill: generate_tests

Описание:
Создает тесты для нового или измененного кода.

Что должен сделать:
- unit tests для service
- integration tests для router / API
- добавить комментарии на русском языке
- не дублировать бессмысленные тесты

---

## Skill: fix_tests

Описание:
Запускает тесты, анализирует падения и исправляет причину.

Порядок:
1. запустить pytest
2. прочитать traceback
3. исправить код или тесты
4. перезапустить тесты
5. убедиться, что решение не ломает архитектуру

---

## Skill: update_architecture

Описание:
Обновляет architecture.md, если структура проекта изменилась.

Когда использовать:
- появился новый модуль
- добавлен новый слой
- изменилась ответственность слоев
- добавлены внешние интеграции

---

## Skill: prepare_commit

Описание:
Готовит изменения к коммиту.

Что должен сделать:
- проверить git diff
- предложить понятное сообщение коммита
- не коммитить с сообщением вида "fix" или "update"
```

### Что делает этот файл

Он превращает твои просьбы в **понятные повторяемые сценарии**.
То есть вместо длинного описания каждый раз ты можешь говорить:

> “Используй Skill: generate_tests для модуля tasks”

# Шаг 6. Заполни `architecture.md`

Это один из самых важных файлов.
Он объясняет Codex, **как именно устроен твой backend**.

```md
# architecture.md

## Цель проекта

Проект представляет собой backend-приложение для управления задачами.
Основная сущность: Task.

---

## Технологии

- Python
- FastAPI
- pytest

---

## Архитектурный стиль

Используется слоистая архитектура.

Поток запроса:
Router -> Controller -> Service -> Repository

---

## Структура каталогов

src/
  app.py                      - точка входа приложения
  modules/                    - доменные модули
  controllers/                - слой контроллеров
  routers/                    - слой роутов
  middleware/                 - middleware приложения
  views/                      - представления / response shaping при необходимости
  interfaces/                 - Pydantic-схемы и интерфейсы данных
  utils/                      - вспомогательные утилиты

tests/
  unit/                       - unit tests
  integration/                - integration tests

---

## Правила проектирования

1. Роуты не должны содержать бизнес-логику.
2. Бизнес-логика должна находиться в service.
3. Repository отвечает только за доступ к данным.
4. Все новые endpoint должны сопровождаться тестами.
5. Все ключевые части кода должны иметь комментарии на русском языке.
6. Если добавляется новая важная подсистема, architecture.md должен быть обновлен.

---

## Модуль tasks

Модуль tasks должен поддерживать:
- создание задачи
- получение списка задач
- получение задачи по id
- обновление задачи
- удаление задачи
- фильтрацию
- пагинацию

---

## Возможное расширение в будущем

- users
- auth
- tags
- notifications
```

### Что делает этот файл

Когда потом попросишь:

> “Создай модуль tasks”

Codex будет опираться не на догадки, а на этот документ.

# Шаг 7. Заполни `testing.md`

```md
# testing.md

## Цель тестирования

Проект должен иметь достаточное покрытие тестами для безопасного рефакторинга и расширения.

---

## Инструменты

- pytest
- httpx / FastAPI test client
- pytest-cov

---

## Типы тестов

### Unit tests
Проверяют:
- service
- utils
- отдельные функции преобразования данных

### Integration tests
Проверяют:
- API endpoints
- корректность статусов
- корректность response body
- фильтрацию
- пагинацию

---

## Требования

1. Каждый новый service должен иметь unit tests.
2. Каждый новый router должен иметь integration tests.
3. Для list endpoint должны быть тесты:
   - без фильтрации
   - с фильтрацией
   - с пагинацией
4. Для create / update / delete endpoint должны быть happy-path тесты.
5. При исправлении бага нужно добавить regression test.
6. Все тесты должны содержать короткие комментарии на русском языке там, где это помогает пониманию.

---

## Целевое покрытие

Минимальное покрытие: 80%
Желаемое покрытие: 90%
```

В чат Codex в VS Code вставь:

```text
Прочитай AGENTS.md, SKILLS.md, CONTROL_FLOW.md, architecture.md и testing.md.
```

3. указываешь сценарий,
4. фиксируешь язык комментариев.

Это сильно улучшает предсказуемость результата.

## 9.2. Команда: создать модуль tasks

```text
Прочитай AGENTS.md, SKILLS.md, CONTROL_FLOW.md, architecture.md и testing.md.

Требования:
- создать repository, service, controller, router и schemas для tasks
- подключить роуты к приложению
- реализовать CRUD

В конце коротко перечисли, какие файлы были созданы.
```

### Что сделает Codex

Обычно после такого промпта он:

* создаст файлы,
* наполнит их кодом,
* подключит роутер,
* добавит тесты.

---

## 9.3. Команда: добавить фильтрацию и пагинацию

```text
Прочитай SKILLS.md, CONTROL_FLOW.md, architecture.md и testing.md.
Действуй как Agent: Backend и Agent: Testing.
Выполни сценарий improve_list_endpoint для модуля tasks.

Требования:
- добавить фильтрацию списка задач
- добавить пагинацию
- обновить схемы, сервис и роутер
- добавить unit и integration тесты для фильтрации и пагинации
- все комментарии в коде писать на русском языке
В конце покажи краткое резюме изменений.
```

### Чему ты здесь учишься

Ты не говоришь “напиши код вот так-то”, ты говоришь **что нужно получить** и **по каким правилам работать**. Это и есть хороший агентный стиль.

---

## 9.4. Команда: найти и починить баги

```text
Действуй как Agent: Maintainer и Agent: Testing.
Используй Skill: fix_tests.

Запусти тесты, найди все падения и исправь причины ошибок.
Если нужно, обнови код, тесты и документацию.
Если исправление влияет на архитектуру, обнови architecture.md.
После исправлений кратко объясни:
- какие тесты падали
- в чем была причина
- что именно ты изменил
```

### Почему это хороший промпт

Он заставляет Codex не просто “чинить”, а **объяснять**, а тебе это важно, потому что ты учишься.

---

## 9.5. Команда: рефакторинг архитектуры

```text
Действуй как Agent: Architect.
Проанализируй текущую структуру проекта и проверь, соответствует ли она architecture.md.

Если найдешь:
- дублирование логики
- смешение слоев
- неправильное размещение файлов
- отсутствие нужных комментариев

то выполни безопасный рефакторинг.
После этого обнови architecture.md при необходимости.
Сначала покажи план изменений, затем внеси изменения.
```

### Здесь важный момент

На архитектурных задачах полезно просить:

> “Сначала покажи план, затем внеси изменения”

Так ты учишься смотреть на ход мысли агента.

---

## 9.6. Команда: подготовить коммит

```text
Действуй как Agent: Maintainer.
Проанализируй текущие изменения в git diff.
Подготовь краткое и понятное сообщение коммита в conventional-style.
Если изменения логически согласованы и тесты проходят, выполни git add и git commit.
Сообщение коммита должно быть осмысленным.
```

### Важно

Я не могу гарантировать, что у тебя Codex в текущей конфигурации сам выполнит именно `git commit` без ограничений среды. Это зависит от настроек доступа, sandbox/rules и того, какие команды ему разрешены. OpenAI отдельно пишет, что rules для управления командами вне sandbox — экспериментальная возможность. ([OpenAI Developers][2])

Поэтому на практике бывает два сценария:

* Codex сам коммитит,
* или Codex готовит message, а коммит делаешь ты.

# Шаг 10. Как связать это с GitHub

## Вариант 2. Codex в GitHub PR

OpenAI пишет, что Codex можно использовать в GitHub для review через комментарий `@codex review` в pull request. ([OpenAI Developers][3])

То есть рабочий учебный процесс может быть такой:

1. Codex в IDE создаёт код.
2. Ты пушишь ветку.
3. Открываешь PR.
4. В PR просишь `@codex review`.

# Шаг 11. Добавь GitHub Actions

OpenAI также публикует Codex GitHub Action `openai/codex-action@v1`, который запускает Codex в CI/CD. ([OpenAI Developers][4])

Но для начала тебе лучше сделать **два workflow**:

1. обычный `pytest`,
2. потом отдельно — Codex Action.

## 11.1. Сначала попроси Codex создать обычный workflow

Промпт в IDE:

```text
Создай GitHub Actions workflow для Python-проекта FastAPI.
Требования:
- запускаться на push и pull_request
- устанавливать зависимости из requirements.txt
- запускать pytest
- при наличии coverage показывать проверку покрытия
Сохрани файл в .github/workflows/tests.yml
Комментарии в YAML не обязательны.
```

## 11.2. Потом отдельный промпт для Codex Action

```text
Создай отдельный GitHub Actions workflow с использованием openai/codex-action@v1.
```

```md
## План: добавить auth

### Цель

Добавить в проект базовую аутентификацию и подготовить архитектуру к разграничению доступа.

### Ожидаемый результат

После выполнения плана проект должен поддерживать:
- регистрацию пользователя
- вход пользователя
- получение токена аутентификации
- защиту приватных маршрутов
- возможность связывать задачи с конкретным пользователем

### Предпочтительный подход

Использовать:
- FastAPI
- Pydantic
- JWT для access token
- отдельный модуль auth
- отдельный модуль users при необходимости

### Этапы

#### Этап 1. Архитектурная подготовка

Что нужно сделать:
- проанализировать текущую архитектуру
- определить место для auth и users
- обновить architecture.md
- при необходимости добавить новые директории и файлы

Ожидаемые изменения:
- новый модуль auth
- возможно новый модуль users
- зависимости между tasks и users должны быть явными и понятными

#### Этап 2. Модели и интерфейсы

Что нужно сделать:
- добавить схемы пользователя
- добавить схемы регистрации и логина
- добавить схему токена
- определить, какие поля обязательны для пользователя

Пример сущностей:
- User
- UserCreate
- UserLogin
- TokenPayload

#### Этап 3. Бизнес-логика auth

Что нужно сделать:
- реализовать регистрацию
- реализовать проверку логина и пароля
- реализовать создание JWT токена
- реализовать получение текущего пользователя из токена

Требования:
- не хранить пароли в открытом виде
- использовать хэширование пароля
- ошибки аутентификации должны быть понятными и безопасными

#### Этап 4. Роуты auth

Что нужно сделать:
- создать auth router
- добавить endpoint регистрации
- добавить endpoint логина
- добавить endpoint получения текущего пользователя при необходимости

Пример маршрутов:
- POST /auth/register
- POST /auth/login
- GET /auth/me

#### Этап 5. Защита маршрутов tasks

Что нужно сделать:
- защитить маршруты задач
- связать задачи с пользователем
- гарантировать, что пользователь работает только со своими задачами

Требования:
- нельзя отдавать задачи другого пользователя
- create/update/delete должны работать в контексте текущего пользователя

#### Этап 6. Тесты

Что нужно сделать:
- добавить unit tests для auth service
- добавить integration tests для auth router
- добавить integration tests для защищенных task routes
- покрыть happy path и ошибки авторизации

Обязательные тесты:
- успешная регистрация
- успешный логин
- неуспешный логин
- доступ к защищенному endpoint без токена
- доступ к защищенному endpoint с токеном
- невозможность читать чужие задачи

#### Этап 7. Документация

Что нужно сделать:
- обновить architecture.md
- обновить testing.md при необходимости
- обновить README.md
- кратко описать auth flow

## План: перевести хранение задач с памяти на БД

- задачи сохраняются в БД
- данные не теряются после перезапуска приложения
- repository работает через БД
- тесты адаптированы под новую модель хранения

### Предпочтительный подход

Базовый вариант:
- SQLite для учебного проекта

Расширяемый вариант:
- PostgreSQL для более production-подхода

### Этапы

- проанализировать текущий repository layer
- определить способ интеграции БД
- обновить architecture.md
- описать новый поток данных

Новая логика:
Router -> Controller -> Service -> Repository -> Database

#### Этап 2. Выбор слоя работы с БД

Что нужно сделать:
- выбрать способ доступа к БД

- при наличии auth предусмотреть user_id

Минимальные поля:
- id
- title
- completed
- created_at

Дополнительно при auth:
- user_id

#### Этап 4. Обновление repository

Что нужно сделать:
- заменить список в памяти на реальные запросы к БД
- реализовать create

- реализовать get by id
- реализовать update
- реализовать delete

Требования:
- код должен быть читаемым

#### Этап 5. Подключение миграций

Предпочтительно:
```
