# PLANS.md

## Общие правила работы с планами

1. Перед началом большой задачи агент обязан прочитать:
   - AGENTS.md
   - SKILLS.md
   - CONTROL_FLOW.md
   - architecture.md
   - testing.md
   - PLANS.md

2. Если задача затрагивает архитектуру, агент должен:
   - сначала показать краткий план изменений
   - затем внести изменения
   - после изменений обновить architecture.md при необходимости

3. Если задача затрагивает тестирование, агент должен:
   - добавить или обновить unit tests
   - добавить или обновить integration tests
   - добавить regression tests для исправленных багов

4. Все комментарии в коде и docstring должны быть на русском языке.

5. После завершения крупной задачи агент должен:
   - кратко перечислить измененные файлы
   - объяснить, что именно было сделано
   - предложить понятное сообщение коммита

## План: добавить auth

### Цель

Добавить в проект базовую аутентификацию и подготовить архитектуру к разграничению доступа.

### Ожидаемый результат

После выполнения плана проект должен поддерживать:
- регистрацию пользователя
- вход пользователя
- получение токена аутентификации
- защиту приватных маршрутов
- возможность связывать задачи с конкретным пользователем

### Предпочтительный подход

Использовать:
- FastAPI
- Pydantic
- JWT для access token
- отдельный модуль auth
- отдельный модуль users при необходимости

### Этапы

#### Этап 1. Архитектурная подготовка

Что нужно сделать:
- проанализировать текущую архитектуру
- определить место для auth и users
- обновить architecture.md
- при необходимости добавить новые директории и файлы

Ожидаемые изменения:
- новый модуль auth
- возможно новый модуль users
- зависимости между tasks и users должны быть явными и понятными

#### Этап 2. Модели и интерфейсы

Что нужно сделать:
- добавить схемы пользователя
- добавить схемы регистрации и логина
- добавить схему токена
- определить, какие поля обязательны для пользователя

Пример сущностей:
- User
- UserCreate
- UserLogin
- TokenPayload

#### Этап 3. Бизнес-логика auth

Что нужно сделать:
- реализовать регистрацию
- реализовать проверку логина и пароля
- реализовать создание JWT токена
- реализовать получение текущего пользователя из токена

Требования:
- не хранить пароли в открытом виде
- использовать хэширование пароля
- ошибки аутентификации должны быть понятными и безопасными

#### Этап 4. Роуты auth

Что нужно сделать:
- создать auth router
- добавить endpoint регистрации
- добавить endpoint логина
- добавить endpoint получения текущего пользователя при необходимости

Пример маршрутов:
- POST /auth/register
- POST /auth/login
- GET /auth/me

#### Этап 5. Защита маршрутов tasks

Что нужно сделать:
- защитить маршруты задач
- связать задачи с пользователем
- гарантировать, что пользователь работает только со своими задачами

Требования:
- нельзя отдавать задачи другого пользователя
- create/update/delete должны работать в контексте текущего пользователя

#### Этап 6. Тесты

Что нужно сделать:
- добавить unit tests для auth service
- добавить integration tests для auth router
- добавить integration tests для защищенных task routes
- покрыть happy path и ошибки авторизации

Обязательные тесты:
- успешная регистрация
- успешный логин
- неуспешный логин
- доступ к защищенному endpoint без токена
- доступ к защищенному endpoint с токеном
- невозможность читать чужие задачи

#### Этап 7. Документация

Что нужно сделать:
- обновить architecture.md
- обновить testing.md при необходимости
- обновить README.md
- кратко описать auth flow

## План: перевести хранение задач с памяти на БД

### Цель

Убрать in-memory хранение задач и заменить его на постоянное хранение в базе данных.

### Ожидаемый результат

После выполнения плана:
- задачи сохраняются в БД
- данные не теряются после перезапуска приложения
- repository работает через БД
- тесты адаптированы под новую модель хранения

### Предпочтительный подход

Базовый вариант:
- SQLite для учебного проекта

Расширяемый вариант:
- PostgreSQL для более production-подхода

Рекомендуемый путь для начала:
1. сначала SQLite
2. потом при необходимости перейти на PostgreSQL

### Этапы

#### Этап 1. Подготовка архитектуры

Что нужно сделать:
- проанализировать текущий repository layer
- определить способ интеграции БД
- обновить architecture.md
- описать новый поток данных

Новая логика:
Router -> Controller -> Service -> Repository -> Database

#### Этап 2. Выбор слоя работы с БД

Что нужно сделать:
- выбрать способ доступа к БД
- создать модуль подключения к БД
- определить базовую конфигурацию

Предпочтительно:
- SQLAlchemy
или
- SQLModel

#### Этап 3. Модель Task в БД

Что нужно сделать:
- создать database model для Task
- определить поля
- синхронизировать model и Pydantic schema
- при наличии auth предусмотреть user_id

Минимальные поля:
- id
- title
- completed
- created_at

Дополнительно при auth:
- user_id

#### Этап 4. Обновление repository

Что нужно сделать:
- заменить список в памяти на реальные запросы к БД
- реализовать create
- реализовать list
- реализовать get by id
- реализовать update
- реализовать delete
- поддержать фильтрацию и пагинацию на уровне repository или service

Требования:
- код должен быть читаемым
- слой repository не должен содержать лишней бизнес-логики

#### Этап 5. Подключение миграций

Что нужно сделать:
- подготовить механизм миграций
- описать начальную схему БД
- добавить команды для локального запуска

Предпочтительно:
- Alembic, если используется SQLAlchemy

#### Этап 6. Обновление тестов

Что нужно сделать:
- убрать зависимость тестов от in-memory хранения
- добавить тестовую БД
- изолировать тестовые данные
- обеспечить повторяемость тестов

Обязательные тесты:
- создание задачи в БД
- получение задач из БД
- обновление задачи
- удаление задачи
- фильтрация
- пагинация

#### Этап 7. Документация

Что нужно сделать:
- обновить architecture.md
- обновить testing.md
- обновить README.md
- описать, как запускать проект с БД
- описать, какие переменные окружения нужны

## Приоритет выполнения планов

Рекомендуемый порядок:
1. перевести хранение задач с памяти на БД
2. затем добавить auth

Причина:
- auth лучше строить уже поверх нормального слоя данных
- связь users и tasks проще реализовать, когда есть БД

## Формат работы агента с большими задачами

Для каждого большого плана агент должен работать так:

1. Сначала показать краткий план реализации.
2. Затем перечислить файлы, которые будут созданы или изменены.
3. Затем внести изменения.
4. Затем создать или обновить тесты.
5. Затем объяснить результат простыми словами.
6. Затем предложить сообщение коммита.
